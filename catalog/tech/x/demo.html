Architecture

⏰ Cron Trigger polling

Basic tier has no push webhooks. A Cron Trigger fires every 5 minutes, fetches new mentions via since_id to avoid reprocessing. Zero gap on Pro with filtered stream.

🔑 OAuth 2.0 PKCE

Posting tweets and DMs requires user context OAuth 2.0 PKCE. Access token cached in KV with auto-refresh 5 minutes before expiry. Bearer token used for read-only searches.

🗃️ KV deduplication

Each processed tweet ID is stored in KV with 7-day TTL. since_id also stored to narrow API queries. Prevents double-replies even if cron overlaps.

🚦 Rate limit awareness

Every response header is checked for x-rate-limit-remaining. On 429, the reset timestamp is logged and processing is skipped — no retry loops that burn quota.

🧵 Thread context

KV context keyed on conversation_id (the root tweet), not individual tweet IDs. Claude sees the full thread when replying deep in a conversation.

🤫 Noise filtering

Query includes -is:retweet -from:ownId. is_echo guard skips own replies. Spam intent classification means the bot never engages with troll mentions.

Create an X Developer App

All API access requires an approved developer account and a project with an App. Basic tier is self-serve; just upgrade via the Developer Portal.

Go to developer.twitter.com → Projects & Apps → New Project
Create an App inside the project — note your Client ID and Client Secret
Under App Settings → User authentication settings — enable OAuth 2.0
Set scopes: tweet.read tweet.write users.read dm.read dm.write offline.access
Set Callback URI to your Worker URL: https://your-worker.workers.dev/callback
Generate a Bearer Token for App-only search (read public tweets without user auth)

Run the OAuth 2.0 PKCE flow

Posting tweets requires a user context token. Run the OAuth flow once to get a refresh token, then the Worker auto-refreshes it via KV.

# 1. Visit the auth URL from your Worker /auth endpoint # Worker generates PKCE verifier + redirects to X # 2. X redirects back to /callback with ?code=... # Worker exchanges code for tokens and stores in KV: x_access_token # expires in ~2h, auto-refreshed x_refresh_token # permanent — store securely in KV
Deploy the Worker first, then visit /auth to kick off the flow
After OAuth, verify with: GET /2/users/me — should return your account
Store your numeric user ID as X_OWN_USER_ID secret to filter your own tweets from mentions

Add secrets and deploy with Cron

Five secrets needed. The wrangler.toml cron schedule controls polling frequency — every 5 minutes is the sweet spot for Basic tier.

# wrangler.toml name = "x-bot" [triggers] crons = ["*/5 * * * *"] # Secrets wrangler secret put BEARER_TOKEN wrangler secret put X_CLIENT_ID wrangler secret put X_CLIENT_SECRET wrangler secret put X_REDIRECT_URI wrangler secret put X_OWN_USER_ID
Deploy: wrangler deploy — Cron starts on first scheduled trigger
Force a test run: wrangler dev --test-scheduled then curl "http://localhost:8787/__scheduled"
Check D1 after first run: wrangler d1 execute DB --command "SELECT * FROM x_mentions"
