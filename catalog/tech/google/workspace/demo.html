Architecture

The bridge. One connection, all APIs.

A service account in a GCP project holds the trust. Domain-wide delegation grants it the right to act on behalf of any user in the Workspace org. The 2nth.ai bridge Worker holds the key — every other skill calls the bridge, never Google directly.

Connection flow

What happens when you connect.

Three steps. The GCP side and the Cloudflare side. Once complete, the bridge is live and every authorised skill can reach the Workspace tenant.

workspace-bridge · Connection wizard
Not connected
1
GCP service account created
A dedicated service account in the 2nth-workspace-bridge GCP project. Its client ID is the identifier Google uses to recognise it in any Workspace tenant.
# Service account identity email: workspace-bridge@ 2nth-workspace-bridge.iam .gserviceaccount.com client_id: "108432917654321098765" project: "2nth-workspace-bridge"
2
Admin grants domain-wide delegation
The Workspace super admin pastes the client ID into admin.google.com → Security → API Controls → Domain-wide Delegation. This authorises the service account to act as any user in the org — the single admin action that unlocks everything.
# admin.google.com → DWD entry client_id: "108432917654321098765" scopes: gmail.readonly gmail.send drive.readonly spreadsheets.readonly calendar.readonly admin.directory.user.readonly
3
Secrets stored in Cloudflare
The service account private key and email are stored as Cloudflare Worker secrets — encrypted at rest, never logged, never exposed. The bridge Worker is deployed and the connection is live.

Admin experience

What the admin actually sees.

From the Workspace admin's perspective, connecting 2nth.ai is a single authorisation action — one entry in the domain-wide delegation list. From that point, they can see exactly what scopes are granted and toggle skills on or off org-wide without touching any code.
