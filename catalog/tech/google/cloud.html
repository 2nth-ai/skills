Google Cloud Platform (GCP)

GCP is the 2nth.ai choice for compute-heavy workloads that don't fit the Cloudflare edge model — containerised legacy apps (ERPNext, Frappe), large relational databases, and ML training. The pattern is: Cloudflare at the edge → GCP for the heavy lift.

Authentication

See tech/google for the shared OAuth 2.0 / service account model.

For GCP APIs specifically, prefer service accounts over user OAuth — GCP resources are not user-owned.

# Authenticate locally
gcloud auth application-default login

# Set project
gcloud config set project YOUR_PROJECT_ID

# Create service account
gcloud iam service-accounts create my-sa \
  --display-name="My Service Account"

# Grant a role
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:my-sa@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/run.invoker"

Cloud Run

Scales to zero. Cold start ~1–2s for Node.js/Python containers. Ideal for on-demand ERPNext instances.

# Deploy a container
gcloud run deploy SERVICE_NAME \
  --image gcr.io/PROJECT_ID/IMAGE:TAG \
  --platform managed \
  --region africa-south1 \
  --allow-unauthenticated \
  --set-env-vars KEY=VALUE \
  --set-secrets KEY=SECRET_NAME:latest \
  --min-instances 0 \
  --max-instances 3 \
  --memory 512Mi \
  --cpu 1

# Get service URL
gcloud run services describe SERVICE_NAME \
  --platform managed --region africa-south1 \
  --format 'value(status.url)'

Scale-to-zero pattern (2nth_erp): Cloud Scheduler sends a request to wake the instance. First request has latency; subsequent requests within the idle timeout (~15min) are warm.

Cloud SQL

Managed MySQL/PostgreSQL. Used for Frappe/ERPNext which requires MySQL 8.0.

# Create instance
gcloud sql instances create INSTANCE_NAME \
  --database-version=MYSQL_8_0 \
  --tier=db-f1-micro \
  --region=africa-south1 \
  --storage-auto-increase \
  --backup-start-time=02:00

# Create database and user
gcloud sql databases create DB_NAME --instance=INSTANCE_NAME
gcloud sql users create DB_USER --instance=INSTANCE_NAME --password=SECRET

# Connect locally via Cloud SQL Auth Proxy
cloud-sql-proxy PROJECT_ID:REGION:INSTANCE_NAME &
mysql -u DB_USER -p -h 127.0.0.1 DB_NAME

Connection from Cloud Run:

# In wrangler.toml / Cloud Run env vars:
DB_HOST=/cloudsql/PROJECT_ID:REGION:INSTANCE_NAME   # Unix socket
DB_SOCKET_PATH=/cloudsql/PROJECT_ID:REGION:INSTANCE_NAME

BigQuery

Serverless SQL analytics. Free tier: 10GB storage + 1TB queries/month.

# Run a query
bq query --use_legacy_sql=false \
  'SELECT client, SUM(amount) as total FROM `project.dataset.invoices` GROUP BY client'

# Load a CSV into a table
bq load --autodetect \
  project:dataset.table \
  gs://bucket/file.csv

# Create a dataset
bq mk --dataset --location=africa-south1 project:dataset_name

From Cloudflare Workers (via REST):

const res = await fetch(
  `https://bigquery.googleapis.com/bigquery/v2/projects/${projectId}/queries`,
  {
    method: 'POST',
    headers: { Authorization: `Bearer ${accessToken}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({
      query: 'SELECT * FROM `project.dataset.table` LIMIT 100',
      useLegacySql: false,
    }),
  }
);

const res = await fetch(

Pub/Sub

Async messaging. Push subscriptions deliver to an HTTP endpoint (Cloud Run, Cloudflare Worker).

# Create topic and subscription
gcloud pubsub topics create my-topic
gcloud pubsub subscriptions create my-sub \
  --topic=my-topic \
  --push-endpoint=https://my-worker.workers.dev/pubsub \
  --ack-deadline=60

# Publish a message
gcloud pubsub topics publish my-topic --message='{"event":"order.created","id":"123"}'

Receiving in Cloudflare Workers:

export default {
  async fetch(request: Request): Promise<Response> {
    const body = await request.json() as { message: { data: string } };
    const payload = JSON.parse(atob(body.message.data));
    // process payload...
    return new Response('OK', { status: 200 }); // must ack within deadline
  }
};

Regions

Prefer africa-south1 (Johannesburg) for SA-based clients. Fallback: europe-west1 (Belgium) for services not yet available in africa-south1.

# List services available in africa-south1
gcloud run regions list

Gotchas

  • africa-south1 gaps: Not all services (e.g. some Vertex AI features) are available in africa-south1 yet — check availability before designing
  • Cloud SQL + Cloud Run: Must add Cloud SQL connection name to Cloud Run service config, not just env var
  • IAM propagation: Role changes take up to 60s to propagate — account for this in CI/CD
  • BigQuery slots vs on-demand: On-demand (pay per query) is fine for ad-hoc; switch to reservations for predictable high-volume workloads
  • Service account keys: Download once, store immediately in Secret Manager — cannot be re-downloaded
