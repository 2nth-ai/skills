💡 The 2nth pattern: Cloudflare Workers handle every incoming message — zero-latency classification with Workers AI, session management in KV, simple replies inline. Connection

Three steps to Azure AI from Workers

An app registration in Entra ID and three Worker secrets. The Worker mints short-lived OAuth 2.0 tokens using client credentials — no user login, no Azure SDK.

2
Store credentials + deploy Azure AI resources
Three secrets in the Worker. The Azure OpenAI endpoint and AI Agent project endpoint go in wrangler.toml as vars (they're not sensitive). Token caching in KV avoids a round-trip on every request.
# Store the three sensitive values as Worker secrets wrangler secret put AZURE_TENANT_ID wrangler secret put AZURE_CLIENT_ID wrangler secret put AZURE_CLIENT_SECRET # wrangler.toml — non-sensitive endpoints [vars] AZURE_OPENAI_ENDPOINT = "https://2nth-ai.openai.azure.com" AZURE_AI_PROJECT_ENDPOINT = "https://2nth-ai-foundry.services.ai.azure.com" AZURE_AGENT_ID = "asst_abc123" # Token with KV caching (avoids Entra roundtrip per request) async function getToken(env: Env, scope: string) { const cached = await env.KV.get(`az_token:${scope}`); if (cached) return cached; const { access_token } = await fetch( `https://login.microsoftonline.com/${env.AZURE_TENANT_ID}/oauth2/v2.0/token`, { method: 'POST', body: new URLSearchParams({ client_id: env.AZURE_CLIENT_ID, client_secret: env.AZURE_CLIENT_SECRET, scope, grant_type: 'client_credentials' }) } ).then(r => r.json()); await env.KV.put(`az_token:${scope}`, access_token, { expirationTtl: 3500 }); return access_token; }

What the Microsoft agentic pattern changes

Without agents
Every AI response is stateless — no memory of the last question
Complex tasks need custom orchestration code in every project
Microsoft 365 data is siloed — no AI can read your email or calendar without bespoke OAuth flows
Teams bots require Bot Framework — heavyweight, slow to build
Document Q&A requires building and maintaining a vector pipeline from scratch
Multi-step tasks (draft → review → send) require manual handoffs
With Azure AI agents
Persistent threads mean the agent remembers context across sessions — stored in Azure, resumed by threadId
Tool calling handles multi-step reasoning — ERP query → analysis → draft → confirm — in a single thread
Graph API as an agent tool gives read access to mail, calendar, Teams without per-user OAuth flows
MCP connector exposes any Worker as a Copilot Studio tool — surfaces in Teams in minutes
AI Agent file_search with Azure AI Search means document Q&A is a tool declaration, not a RAG build
Semantic Kernel process framework handles approval workflows, branching, and multi-agent hand-offs
